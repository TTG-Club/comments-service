# Comments API

REST-сервис комментариев для веб-страниц: древовидные комментарии (корневые + ответы одного уровня), привязка к разделу и URL страницы, аутентификация по JWT (OAuth2 Resource Server).

## Технологический стек

- **Java 21**
- **Spring Boot 4.0.5** — Web MVC, Data JPA, Validation, Security (OAuth2 Resource Server)
- **PostgreSQL** — хранилище данных
- **Liquibase** — миграции схемы БД
- **MapStruct 1.5.5** — маппинг DTO ↔ сущности
- **Lombok**
- **springdoc-openapi** — OpenAPI / Swagger UI
- **Maven** (обёртка `mvnw` в комплекте)

## Возможности

- Постраничная выдача корневых комментариев для пары `section` + `url`.
- Загрузка ответов на конкретный комментарий.
- Подсчёт числа опубликованных комментариев страницы.
- Создание комментариев и ответов, редактирование и «мягкое» удаление (soft delete).
- Автоматический подсчёт числа ответов (`replyCount`) у родителя.
- Проверка прав: редактировать и удалять комментарий может только его автор.
- Нормализация входных данных (`section`/`url` приводятся к нижнему регистру и обрезаются).
- Снимок имени автора на момент создания (`author_name_snapshot`).
- Антиспам-лимит на создание комментариев с нарастающим наказанием (см. ниже).
- Скрытие всех комментариев автора при его блокировке и возврат при разблокировке (см. ниже).

## Модель данных

Таблица `comment.comments`:

| Поле | Тип | Описание |
|------|-----|----------|
| `id` | UUID | Первичный ключ |
| `section` | VARCHAR(100) | Раздел (нормализуется) |
| `url` | VARCHAR(255) | URL страницы (нормализуется) |
| `parent_id` | UUID | Родительский комментарий (`null` — корневой), FK с `ON DELETE CASCADE` |
| `author_id` | UUID | ID автора (из JWT `sub`) |
| `author_name_snapshot` | VARCHAR(255) | Имя автора на момент создания |
| `content` | TEXT | Текст комментария |
| `status` | VARCHAR(50) | Статус (см. ниже) |
| `reply_count` | INT | Число ответов |
| `edited_at` | TIMESTAMPTZ | Время последнего редактирования |
| `deleted_at` | TIMESTAMPTZ | Время удаления |
| `created_at` / `updated_at` | TIMESTAMPTZ | Служебные метки времени |

**Статусы (`CommentStatus`):** `PUBLISHED`, `DELETED`, `PENDING_MODERATION`, `REJECTED`, `SPAM`,
`HIDDEN_BY_BAN`. Выборки возвращают только комментарии со статусом `PUBLISHED`, поэтому любой
другой статус автоматически убирает комментарий со страницы.

`HIDDEN_BY_BAN` — комментарий скрыт из-за блокировки автора (см. [Скрытие комментариев при
блокировке](#скрытие-комментариев-при-блокировке)). В отличие от `DELETED` текст комментария
сохраняется: разблокировка возвращает его как есть.

Таблица `comment.comment_rate_limits` — состояние антиспам-лимита, одна строка на автора:

| Поле | Тип | Описание |
|------|-----|----------|
| `author_id` | UUID | Первичный ключ, автор из JWT `sub` |
| `last_comment_at` | TIMESTAMPTZ | Время последнего **успешно созданного** комментария |
| `violation_level` | INT | Длина текущей серии нарушений |
| `last_violation_at` | TIMESTAMPTZ | Для обнуления серии по времени |
| `blocked_until` | TIMESTAMPTZ | Конец блокировки за спам |
| `updated_at` | TIMESTAMPTZ | Служебная метка времени |

## Конфигурация

Приложение читает настройки из переменных окружения (см. `application.yaml`):

| Переменная | Назначение | По умолчанию |
|------------|-----------|--------------|
| `SPRING_DATASOURCE_URL` | JDBC-URL PostgreSQL | — |
| `SPRING_DATASOURCE_USERNAME` | Пользователь БД | — |
| `SPRING_DATASOURCE_PASSWORD` | Пароль БД | — |
| `DB_POOL_SIZE` | Максимум соединений HikariCP | `10` |
| `DB_MIN_IDLE` | Минимум простаивающих соединений | `2` |
| `COMMENTS_RATE_LIMIT_ENABLED` | Включён ли антиспам-лимит | `true` |
| `AUTH_SERVICE_JWT_SECRET` | HMAC-секрет, которым auth-service подписывает JWT | — |
| `INTERNAL_SERVICE_SECRET` | Общий секрет межсервисных ручек `/api/v1/internal/**` | пусто (ручки закрыты) |

`INTERNAL_SERVICE_SECRET` должен совпадать со значением, настроенным в auth-service, — им
подписываются вызовы скрытия и восстановления комментариев. Если переменная не задана,
внутренние ручки отвечают `401` на любой запрос (fail-closed): не сконфигурированный секрет
закрывает их, а не открывает.

Параметры лимита задаются в `application.yaml` (секция `comments.rate-limit`): `intervals`
(интервалы по уровням нарушений), `block-duration` (длительность блокировки),
`violation-decay` (через сколько без нарушений серия обнуляется).

JWT-аутентификация настроена на выданный локально Authorization Server:

- `issuer-uri`: `http://localhost:8081`
- `jwk-set-uri`: `http://localhost:8081/oauth2/jwks`

> **Внимание:** файл `local.env` в репозитории содержит реальные учётные данные к БД и секреты. Их следует ротировать и не хранить в системе контроля версий — вынесите значения в локальный, неотслеживаемый файл окружения.

## API

Базовый путь: `/api/v1/comments`. Все изменяющие данные операции требуют `Bearer`-токен (JWT).

| Метод | Путь | Описание | Авторизация |
|-------|------|----------|-------------|
| `GET` | `/api/v1/comments?section={section}&url={url}` | Корневые комментарии страницы (постранично, размер по умолчанию 20) | — |
| `GET` | `/api/v1/comments/{parentId}/replies` | Ответы на комментарий | — |
| `GET` | `/api/v1/comments/count?section={section}&url={url}` | Число опубликованных комментариев | — |
| `GET` | `/api/v1/comments/my/count` | Число опубликованных комментариев текущего пользователя (для профиля) | JWT |
| `POST` | `/api/v1/comments` | Создать корневой комментарий | JWT |
| `POST` | `/api/v1/comments/{parentId}/replies` | Создать ответ | JWT |
| `PATCH` | `/api/v1/comments/{commentId}` | Отредактировать свой комментарий | JWT |
| `DELETE` | `/api/v1/comments/{commentId}` | Мягко удалить свой комментарий | JWT |

`author_id` берётся из JWT (`sub`), имя — из клейма `preferred_username` → `username` → `sub`.

### Модераторские эндпоинты

Доступны ролям `MODERATOR` и `ADMIN` (клейм `roles` в JWT). Идут **без** фильтра по статусу:
модератору нужно видеть и `DELETED`, и `HIDDEN_BY_BAN`.

| Метод | Путь | Описание | Авторизация |
|-------|------|----------|-------------|
| `GET` | `/api/v1/comments/moderation` | Все комментарии, от новых к старым | `MODERATOR` / `ADMIN` |
| `GET` | `/api/v1/comments/moderation?authorId={uuid}` | То же, но только комментарии одного автора | `MODERATOR` / `ADMIN` |
| `GET` | `/api/v1/comments/moderation/disliked` | Комментарии с жалобами, от большего числа к меньшему | `MODERATOR` / `ADMIN` |
| `POST` | `/api/v1/comments/{commentId}/restore` | Вернуть удалённый комментарий в опубликованные | `MODERATOR` / `ADMIN` |

`authorId` — необязательный; тот же UUID, что и клейм `sub` в JWT (в админке он показан в
карточке пользователя как «ID (UUID)»). Без него ручка ведёт себя как раньше и отдаёт всю ленту.
В ответе у каждого комментария есть `section` и `url` — по ним строится ссылка на страницу,
где он оставлен.

Восстановление возвращает `DELETED` → `PUBLISHED` и отдаёт `CommentResponse`. Комментарий
в любом другом статусе — `409`: скрытый баном (`HIDDEN_BY_BAN`) снимается разблокировкой автора
в auth-service, а не этой ручкой, иначе следующая же синхронизация вернула бы его в скрытые.
Восстанавливается один узел: ответы под ним остались `PUBLISHED` и снова становятся видны
вместе с ним, потому что обход дерева начинается с корня.

### Межсервисные эндпоинты

Базовый путь: `/api/v1/internal`. Аутентификация не по JWT, а по общему секрету сервисов в
заголовке `X-Service-Token` (значение `INTERNAL_SERVICE_SECRET`) — вызывающему auth-service
пользовательский токен взять неоткуда.

| Метод | Путь | Описание | Авторизация |
|-------|------|----------|-------------|
| `POST` | `/api/v1/internal/comments/hide-by-author` | Скрыть все комментарии автора при блокировке | `X-Service-Token` |
| `POST` | `/api/v1/internal/comments/restore-by-author` | Вернуть комментарии автора при разблокировке | `X-Service-Token` |

```http
POST /api/v1/internal/comments/hide-by-author
X-Service-Token: <INTERNAL_SERVICE_SECRET>
Content-Type: application/json

{ "authorId": "a1c2b3d4-0000-0000-0000-000000000000" }
```

Ответ `200 OK` — сколько комментариев затронула операция:

```json
{ "affected": 3 }
```

Коды ошибок: `401` — секрет неверен или заголовок отсутствует, `400` — некорректное тело
запроса (нет `authorId` или он не UUID).

### Примеры

Создание комментария:

```http
POST /api/v1/comments
Authorization: Bearer <JWT>
Content-Type: application/json

{
  "section": "blog",
  "url": "/posts/hello-world",
  "content": "Отличная статья!"
}
```

Ответ `201 Created`:

```json
{
  "id": "b3f1...",
  "section": "blog",
  "url": "/posts/hello-world",
  "parentId": null,
  "authorId": "a1c2...",
  "authorName": "john",
  "content": "Отличная статья!",
  "status": "PUBLISHED",
  "replyCount": 0,
  "createdAt": "2026-07-15T10:00:00Z",
  "editedAt": null
}
```

## Поведение бизнес-логики

### Ничего не удаляется безвозвратно

Сервис не выполняет физического удаления строк: в коде нет ни одного `DELETE` по таблице
комментариев. Любое «удаление» и «скрытие» — это смена `status`, строка со всем содержимым
остаётся в `comment.comments` навсегда.

| Сценарий | Статус | Что происходит с текстом |
|----------|--------|--------------------------|
| Пользователь удалил свой комментарий | `DELETED` | остаётся в БД |
| Модератор удалил чужой комментарий | `DELETED` | остаётся в БД |
| Модератор восстановил удалённый | `PUBLISHED` | остаётся в БД, `deleted_at` очищается |
| Автор заблокирован | `HIDDEN_BY_BAN` | остаётся в БД |

Публично не отдаётся ничего, кроме `PUBLISHED`: все пользовательские выдачи (список, ответы,
`/count`, `/latest`, `/{id}`) фильтруют по этому статусу, поэтому комментарий с любым другим
статусом со страницы исчезает.

> **Важно для приватности:** списки `/api/v1/comments/moderation` и `/moderation/disliked` идут
> **без** фильтра по статусу, поэтому модератор и администратор видят полный текст удалённых и
> скрытых комментариев. Раньше текст удалённого затирался при удалении и не восстанавливался;
> теперь он сохраняется. Если по требованиям пользователь должен иметь возможность стереть свой
> текст окончательно, для этого нужен отдельный сценарий — мягкое удаление такой гарантии больше
> не даёт.

- **Удаление** — мягкое: статус меняется на `DELETED`, проставляется `deleted_at`, у родителя уменьшается `replyCount`. Текст **не затирается** (см. ниже).
- **Восстановление** удалённого доступно только модератору и администратору (`POST /{commentId}/restore`): статус возвращается в `PUBLISHED`, `deleted_at` очищается, счётчики ответов пересчитываются с нуля. Автор своё удаление отменить не может, разблокировка автора статус `DELETED` тоже не снимает.
- **Ответы** можно оставлять только к опубликованным (`PUBLISHED`) неудалённым комментариям.
- **Редактирование/удаление** чужого комментария приводит к ошибке (`IllegalStateException`).
- Отсутствующий комментарий → `EntityNotFoundException`.

### Скрытие комментариев при блокировке

Блокировкой пользователей заведует auth-service (поле `status`: `ACTIVE` / `BANNED` /
`DELETED`). При бане и разбане он дёргает межсервисные ручки этого сервиса, чтобы комментарии
заблокированного пропали со страниц и вернулись после разблокировки.

- **Скрытие** переводит комментарии автора `PUBLISHED` → `HIDDEN_BY_BAN`. Меняется только статус:
  текст, автор и время создания остаются нетронутыми.
- **Восстановление** переводит `HIDDEN_BY_BAN` → `PUBLISHED`. Комментарии, удалённые самим
  пользователем (`DELETED`) или снятые модератором (`REJECTED`, `SPAM`), не воскресают:
  условие по исходному статусу их не выбирает.
- Обе операции **идемпотентны** по построению: повторный вызов не найдёт подходящих строк и
  вернёт `affected: 0`.

**Счётчики ответов** (`reply_count`, `total_reply_count`) после массовой операции считаются
заново, а не правятся дельтами. Дельты здесь неприменимы: комментарии забаненного автора могут
быть вложены друг в друга, и вычитание `1 + totalReplyCount` по каждому задвоилось бы. Пересчёт
повторяет логику changeSet `backfill-total-reply-count` из миграции `003-total-reply-count.yaml`:
рекурсия идёт только по `PUBLISHED`-узлам, поэтому скрытый узел обрывает путь и его поддерево
уходит из счётчиков предков целиком. Такой пересчёт даёт один и тот же результат независимо от
формы дерева и от числа вызовов, поэтому счётчики после скрытия и последующего восстановления
возвращаются ровно к исходным значениям.

### Антиспам-лимит

Ограничение действует на создание и корневых комментариев, и ответов (общий счётчик — иначе
лимит обходился бы спамом в ответах) и считается по автору из JWT, а не по IP.

| Состояние автора | Минимальный интервал |
|------------------|----------------------|
| Обычное | 20 секунд |
| После нарушения интервала | 60 секунд |
| После второго нарушения подряд | блокировка на 3 часа |

Через **час без нарушений** серия обнуляется и автор возвращается к интервалу 20 секунд.
После блокировки — тоже с чистого листа. Модератор и администратор от лимита освобождены.

Отклонённая попытка **не** сдвигает точку отсчёта: интервал считается от последнего успешно
созданного комментария, иначе непрерывный поток запросов не дал бы автору написать никогда.

Отказ — `429 Too Many Requests` с заголовком `Retry-After` и телом `ProblemDetail`:

```json
{
  "status": 429,
  "detail": "Слишком часто. Следующий комментарий можно отправить через 50 сек.",
  "retryAfterSeconds": 50,
  "blocked": false
}
```

`blocked: true` означает трёхчасовую блокировку за серию нарушений, `false` — просто
невыдержанный интервал. Фронту удобно читать `retryAfterSeconds` для обратного отсчёта.
Лимит обязан жить на сервере: отсчёт в интерфейсе — подсказка пользователю, но `POST`
с валидным токеном можно отправить и мимо интерфейса.
